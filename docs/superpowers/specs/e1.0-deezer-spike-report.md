# E1.0 Spike: Deezer 30s Preview via SvelteKit BFF

**Status:** Proven — integration ready
**Date:** 2026-05-02

## Architecture

BFF (Backend-For-Frontend) proxy pattern:

```
Client (Browser)
  ├─ fetch('/api/track/search?q=...')  →  SvelteKit BFF  →  https://api.deezer.com/search
  ├─ fetch('/api/track/{id}')          →  SvelteKit BFF  →  https://api.deezer.com/track/{id}
  └─ new Audio(preview_url)            →  https://cdns-preview-*.dzcdn.net/...
```

Server-side Deezer calls avoid browser CORS restrictions entirely.

### Key Files

| File | Role |
|------|------|
| `src/routes/api/track/search/+server.ts` | BFF search endpoint with cache + 429 handling |
| `src/routes/api/track/[id]/+server.ts` | BFF track-by-ID endpoint with 429 handling |

| `src/lib/types.ts` | `Track`, `DeezerTrackData`, `MusicProvider` types |

## CORS

**API calls:** Handled by BFF. SvelteKit server-side `fetch()` to `api.deezer.com` has no CORS restrictions.

**Audio playback:** `crossOrigin = 'anonymous'` set on `HTMLAudioElement` in `audio.ts:27`. Deezer CDN (`cdns-preview-*.dzcdn.net`) serves preview MP3s with permissive CORS headers, allowing `Audio` playback and Web Audio API access.

## Rate-Limit Handling

Both BFF endpoints handle Deezer 429 responses:

- `/api/track/search` — Returns 503 with `Retry-After` header forwarded from Deezer
- `/api/track/[id]` — Returns 503 with `Retry-After` header forwarded from Deezer *(fixed in this spike)*

Client should read `Retry-After` header and retry after the specified seconds.

## Caching

Search endpoint only (`/api/track/search`):

- **TTL:** 5 minutes
- **Max entries:** 100
- **Eviction:** FIFO (oldest entry removed when full)
- **Key:** `{query}:{limit}`

Track-by-ID endpoint has no caching (single-track fetches are cheap and infrequent).

## Attribution

`DeezerAttribution.svelte` renders "Powered by Deezer" with optional track link. Component exists and is tested, but not yet wired into any view.
