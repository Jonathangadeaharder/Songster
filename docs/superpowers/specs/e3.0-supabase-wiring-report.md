# E3.0 — Supabase Schema Wiring Report

## Status: COMPLETE

All Supabase schema is fully wired into the client. No blocking gaps found — see noted omissions below.

---

## 1. Browser Client

**File:** `src/lib/supabase.ts`

- Uses `createBrowserClient` from `@supabase/ssr`
- Reads `PUBLIC_SUPABASE_URL` and `PUBLIC_SUPABASE_ANON_KEY` from env
- Exports a singleton `supabase` instance used by all client-side code

## 2. Server Client

**File:** `src/hooks.server.ts`

- Uses `createServerClient` from `@supabase/ssr` with cookie-based auth
- Placeholder mode: detects `https://placeholder.supabase.co` and provides a dummy session for CI/E2E
- Production mode: full cookie auth via `event.locals.supabase` + `safeGetSession()`

## 3. Schema: 5 Tables

| Table | Migration | Realtime | Purpose |
|-------|-----------|----------|---------|
| `rooms` | 001 + 004 | ✓ | Room lifecycle, host, status, winner |
| `players` | 001 | ✓ | Room membership, tokens, host flag |

## 4. Schema: 6 RPCs

| RPC | Migration | Called From |
|-----|-----------|-------------|
| `create_room(host_name)` | 001 | `room.ts:50` |
| `join_room(room_code, player_name)` | 001 | `room.ts:56` |
| `start_game(p_room_code)` | 004 | `room.ts:84` |
| `update_game_state(...)` | 004 | `room.ts:128` |
| `place_card(...)` | 004 | `room.ts:139` |
| `use_token(...)` | 004 | `room.ts:150` |

All RPCs have SECURITY DEFINER. Authorization is enforced via RLS policies (room members can read/write game tables) and in-code checks (start_game, update_game_state, use_token verify caller identity). Note: `place_card` and `join_room` rely solely on RLS for authorization — `place_card` does not verify the caller owns `p_player_id` in-code, which is a known gap since SECURITY DEFINER bypasses RLS.

## 5. RLS Policies (002 + 004)

- All 4 game tables have RLS enabled
- Room members can read rooms, players, game_state, timelines
- Authenticated users can create rooms and join
- Participants can update game_state
- Players can insert own timeline entries
- Players can update own tokens

## 6. Realtime Subscriptions

**File:** `src/lib/room.ts` — `subscribeToRoom()`

Subscribes to 4 tables via `postgres_changes` on a single channel `room:{roomId}`:

| Table | Filter | Events |
|-------|--------|--------|
| `game_state` | `room_id=eq.{roomId}` | `*` (INSERT, UPDATE, DELETE) |
| `players` | `room_id=eq.{roomId}` | `*` |
| `timelines` | `room_id=eq.{roomId}` | `*` |
| `rooms` | `id=eq.{roomId}` | `*` |

The `timelines` filter uses the `room_id` column added in migration 004, enabling efficient per-room filtering without joins.

## 7. Stores: game-remote.ts

**File:** `src/lib/stores/game-remote.ts`

The multiplayer game engine. 16 writable stores mirroring DB state:

| Store | Source | Updated By |
|-------|--------|------------|
| `round` | game_state.round | realtime handler |
| `phase` | game_state.phase | realtime handler |
| `activePlayerId` | game_state.active_player_id | realtime handler |
| `drawPile` | game_state.draw_pile | realtime handler |
| `activeCard` | game_state.active_card | realtime handler |
| `players` | players + timelines join | realtime handler (refetch) |
| `screen` | rooms.status | realtime handler |
| `winner` | rooms.winner_player_id | realtime handler |
| `roomCode`, `myPlayerId`, `isHost`, `connected`, `hoverSlot`, `placedSlot`, `placedResult`, `interceptor`, `dragging` | local/config | client-side |

### Realtime Event Handling

- **game_state changes:** Directly maps payload fields to stores, resets UI state on UPDATE
- **players/timelines changes:** Refetches both tables and rebuilds player objects with timelines
- **rooms changes:** Transitions screen on status change (lobby→play, play→win)

## 9. Minor Notes (Non-blocking)

1. **Duplicate realtime ALTERs:** Migrations 001 and 004 both add game_state/timelines/rooms to `supabase_realtime`. Idempotent — no issue.
2. **timelines.added_at:** Exists in schema but not in `TimelineRow` interface. Not referenced anywhere — no issue.
3. **subscribeToRoom payload type:** Uses `Record<string, unknown>` for `new`. Acceptable since `handleRemoteChange` accesses fields with type assertions.
